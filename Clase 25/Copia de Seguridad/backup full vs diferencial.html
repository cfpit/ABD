Manual Técnico de Respaldos y Contingencias en SQL Server

Estrategia de Seguridad, Cadena de Logs y Database Snapshots

Una arquitectura robusta de disponibilidad y resiliencia en SQL Server combina técnicas tradicionales de backup con tecnologías de congelamiento de estado interno. Su correcta sincronización optimiza el espacio en disco, mitiga errores humanos inmediatos y reduce drásticamente el tiempo de caída.

1. Los Mecanismos de Protección de Datos

Backup Completo (Full)
  • Concepto: Copia el 100% de la base estructural. Es indispensable como base maestra inicial.
  • Uso Operativo: Alta demanda de hardware (I/O, red, CPU). Se programa de manera pasiva (fines de semana).
Backup Diferencial (Differential)
  • Concepto: Respalda únicamente los bloques modificados desde el último backup Full maestro.
  • Ventaja Directa: Agiliza el tiempo diario de copia y simplifica la restauración a solo dos archivos base.
Backup del Log de Transacciones (Transaction Log)
  • Concepto: Grabación secuencial, histórica e ininterrumpida de todas las operaciones atómicas del motor.
  • Ventaja Directa: Permite la restauración a un segundo exacto (Point-in-Time) y recicla el espacio del .ldf.
Requisito: La base de datos debe estar configurada bajo SET RECOVERY FULL
Database Snapshot (Instantánea)
  • Concepto: Vista estática, asíncrona y de solo lectura de la base de datos en un instante de tiempo específico.
  • Mecanismo interno: Utiliza la tecnología Copy-on-Write (Copia al escribir). Inicialmente no ocupa espacio; solo almacena los bloques de datos originales a medida que estos son modificados o eliminados en la base de datos activa.
  • Casos de Uso: Escudo de protección inmediata antes de desplegar scripts masivos riesgosos y aislamiento para reportes pesados de auditoría.
Ubicación obligatoria: Reside estrictamente dentro del mismo servidor de la base original

2. Comparativa Estratégica: Backup vs. Snapshot

Métricas diferenciales clave para el diseño de políticas de contingencia:

Característica Backup Tradicional (.bak / .trn) Database Snapshot (.ss)
Portabilidad Alta. Archivo externo autónomo. Se puede mover a almacenamiento offline u otros servidores. Nula. Depende de la existencia y salud de la base original. Vive y muere en el mismo servidor.
Tiempo de Generación Depende del volumen total de datos (de minutos a horas). Instantáneo. Se crea en milisegundos sin importar el tamaño de la base de datos.
Velocidad de Retorno Lento. Debe leer, copiar y validar los archivos físicos completos en el disco. Ultra rápido. El comando revertir reescribe únicamente los bloques guardados en el diferencial del snapshot.
Log de Transacciones Esencial para mantener la integridad secuencial. No lo utiliza. Ignora los flujos del archivo log.

3. Flujo Técnico Unificado de Código (T-SQL)

Script cronológico completo que asocia modelos de recuperación, copias tradicionales, protección mediante instantánea instantánea y protocolos de recuperación total:

-- ============================================================================
-- PASO 1: CONFIGURACIÓN INICIAL Y GENERACIÓN DE LÍNEAS BASE
-- ============================================================================
ALTER DATABASE [pubs] SET RECOVERY FULL;
GO

BACKUP DATABASE [pubs]
TO DISK = 'c:\backups\pubs.bak'
WITH FORMAT, MEDIANAME = 'Pubs', NAME = 'Full-Pubs backup';
GO

-- Modificación de estructura de datos
USE pubs;
GO
CREATE TABLE prueba(codigo int);
GO

BACKUP DATABASE [pubs]
TO DISK = 'c:\backups\pubs_diff.bak'
WITH DIFFERENTIAL, FORMAT,
MEDIANAME = 'Native_SQLServerDiffBackup', NAME = 'Diff-pubs backup';
GO

-- ============================================================================
-- PASO 2: PROTECCIÓN INMEDIATA (CREACIÓN DEL SNAPSHOT)
-- ============================================================================
-- Creamos la instantánea estática antes de permitir inserciones complejas por la mañana.
-- Nota: Requiere especificar el nombre lógico del archivo de datos ('pubs')
CREATE DATABASE pubs_snapshot_mañana
ON ( 
    NAME = 'pubs', 
    FILENAME = 'c:\backups\pubs_mañana.ss'
)
AS SNAPSHOT OF [pubs];
GO

-- Inserciones de producción por parte de los operadores de la empresa
INSERT INTO prueba VALUES (10);
GO

-- ============================================================================
-- PASO 3: CADENA DE LOGS TRANSACCIONALES
-- ============================================================================
BACKUP LOG [pubs]
TO DISK = 'c:\backups\pubs_log1.trn'
WITH FORMAT, NAME = 'Log-Pubs backup 1';
GO

INSERT INTO prueba VALUES (20);
GO

BACKUP LOG [pubs]
TO DISK = 'c:\backups\pubs_log2.trn'
WITH FORMAT, NAME = 'Log-Pubs backup 2';
GO

-- ============================================================================
-- PASO 4: PROTOCOLO DE CONTINGENCIA A (REVERTIR POR SNAPSHOT - ULTRA RÁPIDO)
-- ============================================================================
-- Escenario: Un script malintencionado corrompe datos y decidimos volver al snapshot de la mañana.
-- Forzamos el modo monousuario para desconectar sesiones activas.
USE master;
GO
ALTER DATABASE [pubs] SET SINGLE_USER WITH ROLLBACK IMMEDIATE;
GO

-- Volcamos el snapshot sobre la base original (revierte los cambios instantáneamente)
RESTORE DATABASE [pubs] 
FROM DATABASE_SNAPSHOT = 'pubs_snapshot_mañana';
GO

ALTER DATABASE [pubs] SET MULTI_USER;
GO

-- Una vez validada la estabilidad, destruimos el snapshot (Buena práctica de rendimiento)
DROP DATABASE pubs_snapshot_mañana;
GO

-- ============================================================================
-- PASO 5: PROTOCOLO DE CONTINGENCIA B (RESTAURACIÓN TRADICIONAL COMPLETA)
-- ============================================================================
-- Escenario alternativo: Fallo físico de disco. Reconstrucción completa mediante archivos .bak y .trn.
RESTORE DATABASE [pubs]
FROM DISK = 'c:\backups\pubs.bak'
WITH NORECOVERY, REPLACE;
GO

RESTORE DATABASE [pubs]
FROM DISK = 'C:\backups\pubs_diff.bak'
WITH NORECOVERY;
GO

RESTORE LOG [pubs]
FROM DISK = 'c:\backups\pubs_log1.trn'
WITH NORECOVERY;
GO

RESTORE LOG [pubs]
FROM DISK = 'c:\backups\pubs_log2.trn'
WITH RECOVERY;
GO

4. Auditoría e Inspección de Metadatos

Comando para evaluar las rutas y nombres lógicos contenidos en un backup sin ejecutar operaciones de restauración:

RESTORE FILELISTONLY FROM DISK = 'C:\backups\pubs.bak';

Estrategia de Arquitectura Unificada

Utilice las copias de seguridad tradicionales (Full, Diferenciales y Logs) como su salvaguarda inquebrantable frente a desastres físicos, pérdidas de hardware o corrupción total de servidores. En paralelo, incorpore los Database Snapshots como herramientas tácticas operativas de ejecución inmediata para proteger el entorno antes de cambios críticos de software o actualizaciones masivas en la base de datos de producción.
